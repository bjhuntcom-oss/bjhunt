## 1. CHAT SSE — bugs reproductibles

### CHAT-P0-1 — Course double-send / changement de conversation pendant streaming
- **Fichiers** : `app/[locale]/dashboard/chat/page.tsx:659, 906-919`
- **Cause** : la dédup `requestIdRef.current !== currentRequestId` (l.659) protège l'enqueue mais pas la détection delta vs cumulatif (l.906-919) qui compare à `lastAiContentRef.current` sans re-vérifier l'identité du run.
- **Repro** : 1) lance un stream. 2) clique une autre conversation pendant le streaming. 3) envoie un nouveau message dans la nouvelle. 4) le stream 1 termine et appende ses tokens dans le message 2.
- **Impact** : corruption silencieuse de l'historique chat.

### CHAT-P0-2 — Prepare succeed → stream fail = `agent_runs` orphelins
- **Fichiers** : `backend/src/routes/chat.ts:176-179, 207, 282-286`
- **Cause** : `POST /prepare` insère le user message + crée `agent_run` (status=`running`). Si le client n'ouvre jamais le SSE (close tab), aucun cleanup. Pas de timeout query, pas de cron côté repo.
- **Repro** : POST /prepare → close tab. `SELECT … FROM agent_runs WHERE status='running' AND completed_at IS NULL` accumule indéfiniment.
- **Impact** : DB pollution + impact billing si jamais le quota lit `agent_runs.status`.

### CHAT-P0-3 — Injection SSE par event LangGraph contenant `\n`
- **Fichiers** : `backend/src/routes/chat.ts:429`, `app/[locale]/dashboard/chat/page.tsx:747`
- **Cause** : sanitize `replace(/[\r\n]/g, "")` n'est appliqué que dans la branche emit directe. Le parser frontend (l.747) extrait `event:` sans re-sanitiser. Un event LangGraph forgé `event: "token\ndata: injected"` contourne le filtre.
- **Impact** : injection SSE / confusion d'event types.

### CHAT-P1-1 — Détection cumulatif vs delta : final chunk dropé
- **Fichiers** : `backend/src/lib/langgraph-sse-transform.ts:136`, `page.tsx:908`
- **Cause** : si chunk incoming `=== state.fullResponse`, delta="" silencieusement. Si Ollama réémet le final chunk, second drop côté frontend (`current.endsWith(incoming)`).
- **Impact** : token count audit faux. Pas critique fonctionnellement.

### CHAT-P1-2 — Agent picker out-of-sync au load history
- **Fichiers** : `page.tsx:385, 594, 1406`
- **Repro** : conversation persistée avec `agentGraph="recon"`, état React `selectedAgent="analyst"`. User tape Entrée avant rerender → message envoyé avec `analyst`.

### CHAT-P1-3 — Ticket TTL 2min expire pendant le stream
- **Fichiers** : `backend/src/lib/stream-ticket.ts:29`, `chat.ts:258-286`
- **Repro** : POST /prepare → tab background 120s+ → foreground → 401, aucun retry/refresh logic. User perd silencieusement la réponse.

### CHAT-P1-4 — Reader/EventSource non fermé sur exception avant `finally`
- **Fichiers** : `page.tsx:634-697` (interval), `479` (releaseLock)
- **Impact** : leak mémoire progressif sur session longue.

### CHAT-P1-5 — `on_disconnect: "cancel"` ne propage pas l'abort
- **Fichiers** : `backend/src/lib/langgraph-client.ts:110`
- **Cause** : SSE n'expose pas le close client à LangGraph. Run continue côté backend.
- **Impact** : compute gaspillé, fraude possible (free tier qui spam-abort).

### CHAT-P1-6 — Cascade delete conversation incomplète
- **Fichier** : `chat.ts:1127` (et schéma)
- **Cause** : DELETE soft (`status='deleted'`). `chat_messages`, `agent_runs`, `file_uploads` restent.
- **Impact** : storage leak + GDPR (RGPD effacement non effectif).

### CHAT-P1-7 — Pas de rate-limit sur POST /prepare quand Redis down
- **Fichier** : `chat.ts:55-66`, `middleware/rate-limit.ts:47-48`
- **Cause** : fail-open quand Redis HS. `pendingStreams` Map non bornée.
- **Impact** : DoS interne (spam → OOM backend).

### CHAT-P1-8 — Tool call ID collision sur streams concurrents
- **Fichiers** : `langgraph-sse-transform.ts:172`, `page.tsx:793-802`
- **Cause** : `next.set(tc.id, …)` overwrite si deux streams (modèles GLM/Qwen réutilisent ids).

### CHAT-P1-9 — `setInterval` speed counter jamais clear sur exception
- **Fichier** : `page.tsx:564-573, 690-699, 758`
- **Impact** : tokens/sec UI corrompu après un parse error.

### CHAT-P1-10 — Events SSE inconnus silencieusement dropés
- **Fichier** : `page.tsx:768-1066`
- **Cause** : switch sans default. Tout event futur (LangGraph upgrade) → silent loss.

### CHAT-P1-11 — Race conversationId au switch rapide
- **Fichiers** : `page.tsx:340, 596-597, 631`
- **Repro** : click engagement A → click B avant load → send → réponse de B mappée à A par `returnedConvId`.

## 2. AUTH / 2FA

### AUTH-P1-1 — Email verification jamais enforcée
- **Fichiers** : `backend/src/routes/auth.ts:53-121` (register crée session direct), `schema.sql:32` (`email_verified` colonne existe mais zéro check)
- **Repro** : register `attacker@victim-typo.com` → login direct → accès dashboard sans owning email.
- **Impact** : ATO sur typo, spam comptes, abus free tier.

### AUTH-P1-2 — Change password n'invalide pas autres sessions
- **Fichier** : `auth.ts:403-431`
- **Cause** : update `password_hash` seul. Cookies volés restent valides 30j.
- **Fix** : `deleteUserSessions(userId)` après update (sauf session courante).

### AUTH-P1-3 — Backup code 2FA consommation non atomique
- **Fichier** : `routes/two-factor.ts:227-233`
- **Cause** : verify+remove en deux étapes (SELECT puis UPDATE). Deux requêtes simultanées avec même backup code passent toutes les deux.
- **Fix** : `UPDATE … SET totp_backup_codes = array_remove(…) WHERE id=$1 AND $2 = ANY(totp_backup_codes) RETURNING …` atomique.

### AUTH-P1-4 — TempToken 2FA replayable
- **Fichiers** : `auth.ts:183-194`, `two-factor.ts:200-207`
- **Cause** : tempToken = `sha256(userId:secret:expiry):userId:expiry`. Pas de nonce, pas de tracking single-use. Brute-force TOTP unlimited tant que tempToken valide.

### AUTH-P1-5 — Lockout contournable via rotating IP
- **Fichier** : `auth/lockout.ts:39-42`
- **Cause** : key = `(email, ip)` sans cap absolu par email. 10 IPs × 5 essais = 50 essais/h sur même email.
- **Fix** : ajouter contre-clé `(email)` avec seuil plus large (ex 30/h).

### AUTH-P2-1 — CSRF accepte Origin manquant si session présente
- **Fichier** : `middleware/csrf.ts:33-44`
- **Note** : SameSite=Lax limite déjà l'attaque, mais durcir.

### AUTH-P2-2 — Logout = single session (pas multi-device)
- **Fichier** : `auth.ts:232-253` — pas d'option "logout all devices".

### AUTH-P2-3 — Reset email pas rate-limité par user
- **Fichier** : `auth.ts:305-345` — global rate-limit auth seulement.

### AUTH-P2-4 — Backup code entropy ~47 bits
- **Fichier** : `auth/totp.ts:89` — `nanoid(8).replace(/[^a-z0-9]/g, "x")` réduit l'entropie.

### AUTH-P2-5 — Pas de notif email sur change password
- **Fichier** : `auth.ts:403-431` — recommandé pour audit trail.

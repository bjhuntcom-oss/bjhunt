## 2. Pourquoi le chat ne fonctionne pas — réponse définitive

Après ~15 itérations de fix infructueuses, la cause racine est identifiée avec **évidence reproductible**. Ce n'est PAS ce que `docs/CHAT-DEBUG-PROMPT.md` supposait (Vercel 10s timeout + cache CORS 24h). C'est **deux bugs additifs** que les fix précédents n'ont jamais traités :

### 2.1 Bug A — Backend Bun ne flush pas (cause 502) [Finding #2-1]

`backend/src/routes/chat.ts` construit la réponse SSE via :
```ts
return new Response(new ReadableStream({ start(controller) { ... } }), {
  headers: { "Content-Type": "text/event-stream", ... }
});
```

**Sur Bun**, `new Response(ReadableStream)` *batch* les chunks au lieu de flusher chaque événement. Les octets s'accumulent dans les buffers kernel jusqu'à ce que LangGraph ferme la connexion amont (10 s plus tard). À ce moment, **Caddy voit `unexpected EOF`** et retourne 502 au client.

**Preuves live (logs VPS)** :
- Backend : `GET /api/chat/stream/5de42e43-… 200 6ms` ← la "réponse" Hono est construite en 6 ms alors que le run prend 10 s.
- LangGraph : `Background run succeeded run_exec_ms=9736` ← le moteur marche.
- LiteLLM : HTTP 200 d'Ollama Cloud ← LLM OK.
- Caddy : **17 entrées `net/http: HTTP/1.x transport connection broken: unexpected EOF` + 2 `aborting with incomplete response`** dans les 3000 dernières lignes, toutes sur `/api/chat/stream*`, toutes terminant à 8-12 s en HTTP 502.

**Fix** : migrer vers le helper `streamSSE` de Hono ([hono.dev/docs/helpers/streaming](https://hono.dev/docs/helpers/streaming)) qui force le flush après chaque write. Voir bug GitHub oven-sh/bun#13923.

### 2.2 Bug B — Frontend parser non-spec-compliant (cause "réponse vide") [Finding #1-1]

`app/[locale]/dashboard/chat/page.tsx:608` implémente un parser SSE manuel :
```ts
buffer += decoder.decode(value, { stream: true });
const blocks = buffer.split("\n\n");          // ← casse sur CRLF
buffer = blocks.pop() ?? "";
for (const block of blocks) {
  const lines = block.split("\n");            // ← laisse \r dans les lignes
  ...
  JSON.parse(dataLines.join("\n"));           // ← fail car "{...}\r{...}\r"
}
```

**LangGraph (Python starlette) émet des frames SSE terminés par `\r\n\r\n`** — ce qui est valide selon HTML Living Standard §9.2.6 (CR, LF et CRLF sont tous autorisés). Le parser frontend splitte seulement sur `\n\n`, laissant des `\r` dans les lignes et collant deux frames JSON en un seul string. `JSON.parse` échoue avec `SyntaxError: Unexpected non-whitespace character after JSON`.

Et comme le `catch { return; }` ligne 703 **avale silencieusement toutes les erreurs**, aucun frame ne parse jamais, `lastAiContentRef` reste vide, le `finally` flip `emptyResponse: true` → l'utilisateur voit "Le moteur IA est temporairement indisponible".

**Preuve live** (harness `browser_evaluate` dans Playwright) :
```
bodyLen=4294, numBlocks=3, parsedOk=0,
parseErrors=[{event:"values", err:"SyntaxError: Unexpected non-whitespace character after JSON at position 63"}, ...]
```

**Fix 1 ligne** :
```ts
// chat/page.tsx, dans la boucle reader (~ligne 607)
buffer += decoder.decode(value, { stream: true });
const normalised = buffer.replace(/\r\n/g, "\n");  // AJOUTER CETTE LIGNE
const blocks = normalised.split("\n\n");
buffer = blocks.pop() ?? "";
```

### 2.3 Mesures additionnelles recommandées (post-fix)

- **Finding #1-3** : `stream_mode: ["values", "custom"]` est demandé à LangGraph — jamais `"messages"`. La branche `case "token":` du frontend est du code mort. Pour du streaming token-par-token, il faut demander `"messages"` explicitement.
- **Finding #1-11** : Le cache-buster `_t=Date.now()` est inefficace — Chrome cache les preflights **sans la query string** et la borne supérieure de `Access-Control-Max-Age` est 7200 s (2h), pas 24h.
- **Finding #2-4** : `pendingStreams` est une `Map` in-process → casse au restart + ne scale pas horizontalement. À migrer vers Redis.
- **Finding #1-16** : Le fallback "emptyResponse" masque les erreurs parser → doit logger le raw body avant de flip le banner.
